## Persona 1 — นพ. ธนกฤต วงศ์อนันต์
### "เวลาคนไข้สำคัญกว่าเวลาพิมพ์"

> รูปแทน: ชายไทย แต่งกายเสื้อกาวน์ ยืนข้างเตียงคนไข้ ถือแท็บเล็ตในมือเดียว

---

### 1. Demographic & Background

| หัวข้อ | รายละเอียด |
|---|---|
| **ชื่อ-นามสกุล** | นายแพทย์ ธนกฤต วงศ์อนันต์ ("หมอเอก") |
| **อายุ** | 42 ปี |
| **ตำแหน่ง** | อายุรแพทย์โรคหัวใจ (Cardiologist) / แพทย์ผู้เชี่ยวชาญ |
| **แผนก** | อายุรกรรม (Med) — ออกตรวจ OPD 2 วัน/สัปดาห์, ราวด์ IPD ทุกเช้า, อยู่เวร on-call |
| **ประสบการณ์วิชาชีพ** | 16 ปี (จบแพทย์ + เรียนต่อเฉพาะทาง 5 ปี) — เชี่ยวชาญสูง มั่นใจในการวินิจฉัย |
| **Tech Savviness** | ★★★☆☆ (ปานกลาง–ค่อนข้างสูง) ใช้ iPhone/iPad คล่อง ใช้แอปดูบทความ UpToDate, ดูผล Lab ผ่านมือถือ แต่ "ไม่อดทน" กับซอฟต์แวร์ที่ช้าหรือออกแบบแย่ — ถ้ากดแล้วไม่เข้าใจใน 3 วินาทีจะเลิกใช้แล้วโทรสั่งพยาบาลแทน |

**ลักษณะนิสัยที่ส่งผลต่อการใช้งาน:**
- คิดเร็ว ทำเร็ว ใจร้อน ทนกับ workflow ที่ "คลิกเยอะ" ไม่ได้
- ให้คุณค่ากับความถูกต้องทางคลินิกสูงมาก — ยอมเสียเวลาตรวจทานถ้าเรื่องนั้นเกี่ยวกับความปลอดภัยคนไข้
- มองระบบ IT เป็น "เครื่องมือ" ไม่ใช่ "งานเอกสาร" ถ้ามันทำให้เขาเป็นหมอที่ดีขึ้นเขาจะรัก ถ้ามันทำให้เขาเป็นเสมียนเขาจะเกลียด

---

### 2. Daily Workflow & Environment

**ภาพหนึ่งวันของหมอเอก:**

- **07:30** ราวด์วอร์ด IPD — เดินดูคนไข้ในทีละเตียง 12–18 คน มีเวลาเฉลี่ย **3–5 นาที/เตียง** ต้องดูประวัติ, ผล Lab เมื่อคืน, vital sign trend, แล้วสั่งการรักษาทันทีข้างเตียง
- **09:00** ออก OPD — คนไข้ลงทะเบียนรอ **40–60 คน** ต่อบ่าย เวลาจริงต่อคน **4–7 นาที** ต้องซักประวัติ ตรวจร่างกาย วินิจฉัย สั่งยา/Lab/นัด ภายในเวลานั้น
- **ระหว่างวัน** ถูกขัดจังหวะตลอด (โทรปรึกษาจากวอร์ดอื่น, พยาบาลมาถามเรื่องยา, ญาติขอคุย)
- **บ่าย–เย็น** ตามผล, เซ็นเอกสารค้าง, สรุปเวชระเบียนที่ยังไม่ปิด (งานที่เขา "เกลียดที่สุด")

**อุปกรณ์หลักที่ใช้เปิด CIS:**

| สถานการณ์ | อุปกรณ์ | ลักษณะการใช้ |
|---|---|---|
| ราวด์ IPD | **แท็บเล็ต (iPad)** ถือมือเดียว / Computer on Wheels (CoW) | ยืน เดิน มือข้างหนึ่งอาจถือ stethoscope — ต้องการ touch target ใหญ่, อ่านง่ายในแสงวอร์ด |
| OPD | **คอม All-in-One บนโต๊ะตรวจ** + จอที่สอง | นั่งหน้าจอ พิมพ์เร็ว สลับหน้าจอบ่อย ต้องการ keyboard shortcut |
| ปรึกษา/นอกเวลา | **มือถือส่วนตัว** | ดูผล Lab / รูป / สรุปเคสแบบ read-only |

> **Insight:** หมอไม่ได้ใช้ระบบจากที่เดียว — เขา "ย้ายร่าง" ระหว่างจอใหญ่กับแท็บเล็ตตลอดวัน UI ต้อง responsive จริง ไม่ใช่แค่ย่อจอ

### 3. Goals & Motivations

**เป้าหมายรายวัน:**
- ดูคนไข้ให้ครบทุกคน "ทันเวลา" โดยไม่พลาดข้อมูลสำคัญ (ประวัติแพ้ยา, ผลผิดปกติ, แนวโน้มทรุด)
- ตัดสินใจทางคลินิกได้เร็วและมั่นใจ — "เห็นภาพรวมคนไข้ในหน้าเดียว" ภายในไม่กี่วินาที
- สั่งยา/Lab/นัด ให้จบในคลิกน้อยที่สุด
- ปิดเวชระเบียนให้ครบ ไม่มีงานค้างกลับบ้าน

**ความคาดหวังสูงสุดต่อ CIS:**
> "ขอแค่เปิดมาแล้วเห็นเลยว่าคนไข้คนนี้เป็นอะไร เคยเป็นอะไร กินยาอะไรอยู่ — แล้วให้ผมสั่งของได้ไว ๆ ไม่ต้องไล่กดทีละหน้า"

- **One-screen summary** — ประวัติย่อ + ปัญหาปัจจุบัน + ยาเดิม + allergy + ผล Lab ล่าสุด ในหน้าจอเดียว
- **Trend ไม่ใช่ตัวเลขเดี่ยว** — อยากเห็นกราฟ vital/lab มากกว่าเลขในตาราง
- **สั่งของเร็ว** — order set, ยาที่สั่งบ่อย, สั่งซ้ำจากครั้งก่อนได้ในคลิกเดียว
- **ระบบจำสิ่งที่เขาทำซ้ำ ๆ ได้** (favorite, last used)

### 4. Pain Points & Frustrations

**สิ่งที่ทำให้หงุดหงิดที่สุด:**

- 🔴 **ระบบช้า / โหลดนาน** — แค่หน้าเดียวโหลด 5–10 วินาที × คนไข้ 50 คน = เวลาหายไปเป็นชั่วโมง
- 🔴 **คลิกเยอะกว่าจะสั่งยาได้** — เลือกยา → ขนาด → วิธีใช้ → จำนวน → ยืนยัน ผ่าน 5–6 ป๊อปอัป
- 🔴 **ตัวหนังสือเล็ก จอรก** — ข้อมูลอัดแน่นจนหาสิ่งที่ต้องการไม่เจอ ต้องเพ่ง
- 🔴 **ข้อมูลกระจัดกระจาย** — ประวัติแพ้ยาอยู่แท็บนึง ยาเดิมอีกแท็บ Lab อีกหน้า ต้องสลับไปมา
- 🔴 **เด้งเตือนพร่ำเพรื่อ (alert fatigue)** — pop-up เตือนจนชินแล้วกดปิดอัตโนมัติ ทำให้พลาดเตือนที่สำคัญจริง

**Time Pressure & Cognitive Load:**
- มีเวลา **4–7 นาที/คนไข้** ต้องทั้งฟัง ตรวจ คิด และพิมพ์ไปพร้อมกัน
- ภาระสมองสูง: ต้องจำเคสคนก่อนหน้า + คิดเคสคนปัจจุบัน + รับสายปรึกษา
- ความกลัวที่ฝังอยู่ตลอด: **"พลาดข้อมูลสำคัญแล้วคนไข้เป็นอันตราย"** → ทุกการออกแบบที่เพิ่มโอกาสพลาด คือความเครียด

### 5. Specific Feature Interaction — AI Summary & Body Map

#### 🤖 ระบบสรุปการซักประวัติด้วย LLM

**ทัศนคติ:** *เปิดรับแต่ระแวง (cautiously optimistic)*
> "ถ้ามันช่วยร่างให้ผมแก้ ผมโอเคมาก แต่ถ้ามันสรุปผิดแล้วผมเซ็นไปโดยไม่ทันดู — นั่นคือชื่อผมที่ติดอยู่ในเวชระเบียน"

**ความคาดหวัง:**
- AI ร่าง SOAP / progress note ให้ แล้วเขา **แก้ไขและอนุมัติ** (human-in-the-loop เสมอ)
- สรุปต้อง **อ้างอิงได้** — แตะประโยคในสรุปแล้วเห็นว่ามาจากบทสนทนาช่วงไหน (traceability)
- ต้อง **เร็วกว่าการพิมพ์เอง** ไม่งั้นไม่คุ้ม

**ความกลัว (Fears):**
- 😨 **AI สรุปผิด / แต่งข้อมูลที่ไม่มี (hallucination)** โดยเฉพาะตัวเลข ขนาดยา ค่าผล
- 😨 **เสียเวลาแก้มากกว่าพิมพ์เอง** — ถ้าต้องอ่านทวนทั้งย่อหน้าเพื่อจับผิด สู้พิมพ์เองดีกว่า
- 😨 **ความรับผิดทางกฎหมาย** — ใครรับผิดถ้า AI ผิดแล้วเขาเซ็น

**สิ่งที่อยากเห็นในหน้าจอ (เพื่อความปลอดภัย + เร็ว):**
- ✅ ป้ายชัดเจนว่า **"ร่างโดย AI — รอแพทย์ยืนยัน"** ห้าม auto-save เข้าเวชระเบียนเด็ดขาด
- ✅ **Highlight ส่วนที่ AI ไม่มั่นใจ** หรือเป็นตัวเลข/ยา ให้ตรวจเป็นพิเศษ
- ✅ แก้ไขแบบ **inline** ได้ทันที ไม่ต้องเปิดหน้าต่างใหม่
- ✅ ปุ่ม **"ยืนยัน"** เด่นชัด แยกออกจาก "แก้ไข" — และ diff ให้เห็นว่าเปลี่ยนอะไรจากต้นฉบับ

#### 🧍 Human Anatomy Indicator (Body Map แบบ Interactive)

**ทัศนคติ:** *ชอบมากในฐานะเครื่องมือ "อ่านเร็ว"*
> "ถ้าผมเหลือบดูรูปคนแล้วเห็นจุดแดงตรงหน้าอกกับเข่าซ้าย ผมเข้าใจเคสได้ใน 1 วินาที เร็วกว่าอ่านข้อความ 3 บรรทัด"

**ความคาดหวัง:**
- ใช้เป็น **navigation + visualization** — แตะอวัยวะเพื่อกระโดดไปดู note/ภาพของจุดนั้น
- แสดง **ตำแหน่งอาการ/รอยโรค** บนร่างกายแบบเห็นภาพรวมทันที (เจ็บที่ไหน, แผลที่ไหน)
- ต้อง **ตรงตำแหน่งจริง** — ซ้าย/ขวาต้องไม่สลับ (ผิดข้างคือเรื่องใหญ่ในทางคลินิก)

**ความกลัว:**
- 😨 **Body map สลับซ้าย-ขวา** หรือชี้ผิดตำแหน่ง → นำไปสู่ความเข้าใจผิดทางคลินิก
- 😨 เป็นแค่ "ของเล่นสวย ๆ" ที่ช้าและกดยาก ไม่ได้ช่วยงานจริง
- 😨 กดบนแท็บเล็ตขณะยืนราวด์แล้ว **กดพลาดจุด** (touch target เล็ก)

**สิ่งที่อยากเห็น:**
- ✅ ระบุชัดว่ามองจาก **มุมไหน** (ด้านหน้า/หลัง) และซ้าย-ขวาเป็นมุมของคนไข้หรือผู้ดู
- ✅ **อ่านได้เร็วแบบ glanceable** — สี/ความเข้มบอกระดับความรุนแรง
- ✅ touch target ใหญ่พอสำหรับนิ้วบนแท็บเล็ต และทำงานได้ทั้ง mouse + touch
- ✅ เป็น **input ที่เร็ว** — แตะจุดแล้วเลือกอาการจาก list สั้น ๆ ไม่ใช่พิมพ์เอง

## Persona 2 — คุณ ศิริพร ใจดีงาม
### "พยาบาลคือคนที่กรอกข้อมูลให้ทั้งโรงพยาบาล"

> รูปแทน: หญิงไทย ชุดพยาบาล ยืนหลังเคาน์เตอร์พยาบาล มีคนไข้ต่อแถวรออยู่

---

### 1. Demographic & Background

| หัวข้อ | รายละเอียด |
|---|---|
| **ชื่อ-นามสกุล** | นางสาว ศิริพร ใจดีงาม ("พี่ศิ") |
| **อายุ** | 34 ปี |
| **ตำแหน่ง** | พยาบาลวิชาชีพชำนาญการ (Registered Nurse) |
| **แผนก** | OPD อายุรกรรม (หมุนเวียนช่วยเวร IPD) — ทำหน้าที่คัดกรอง (triage) + ซักประวัติเบื้องต้น + จัดคิวตรวจ |
| **ประสบการณ์วิชาชีพ** | 11 ปี — เชี่ยวชาญหน้างาน รู้ flow โรงพยาบาลทุกซอก คือคนที่ระบบ "พึ่งพา" ในการกรอกข้อมูลส่วนใหญ่ |
| **Tech Savviness** | ★★★☆☆ (ปานกลาง) ใช้ระบบ CIS/HOSxP มาหลายปีจนคล่อง "ตามที่เคยทำ" จำลำดับการกดเป็นกล้ามเนื้อ (muscle memory) แต่ปรับตัวกับ UI ใหม่ได้ช้ากว่า — กลัวกดผิดแล้วข้อมูลคนไข้เสียหาย |

**ลักษณะนิสัยที่ส่งผลต่อการใช้งาน:**
- ละเอียด รอบคอบ รับผิดชอบสูง — เป็นด่านหน้าที่ข้อมูลผิดจะส่งต่อให้หมอผิดทั้งสาย
- ทำงานภายใต้แรงกดดันสองทาง: **คนไข้หน้าเคาน์เตอร์** ที่รอและบ่น + **หมอ** ที่รอข้อมูล
- คุ้นกับ keyboard + การพิมพ์เร็ว มากกว่า touch/gesture — ชอบ flow ที่ "เดาได้" และเหมือนเดิมทุกครั้ง

---

### 2. Daily Workflow & Environment

**ภาพหนึ่งวันของพี่ศิ:**

- **07:00** เปิดเคาน์เตอร์ คนไข้เริ่มต่อแถวคัดกรอง — วัด vital sign, ชั่งน้ำหนัก, ซักอาการนำ, คัดกรองความเร่งด่วน
- **ทั้งเช้า** กรอกข้อมูลคนไข้ **เฉลี่ย 1.5–3 นาที/คน** เป็นร้อยคน — ป้อน vital, อาการสำคัญ (CC), ประวัติแพ้ยา, ส่งคิวเข้าห้องตรวจหมอ
- **ระหว่างนั้น** รับโทรศัพท์, ตอบคำถามคนไข้/ญาติ, ประสานห้อง Lab/X-ray, จัดคิว, ดูแลคนไข้ฉุกเฉินแทรก
- **ช่วงบ่าย** ติดตามผล, บันทึกการพยาบาล, เคลียร์เอกสารส่งต่อ

**อุปกรณ์หลักที่ใช้เปิด CIS:**

| สถานการณ์ | อุปกรณ์ | ลักษณะการใช้ |
|---|---|---|
| คัดกรอง/ลงทะเบียน | **คอม All-in-One บนเคาน์เตอร์** + คีย์บอร์ด | นั่ง/ยืนที่เคาน์เตอร์ พิมพ์เร็วมาก ใช้ keyboard เป็นหลัก คนไข้ยืนรอตรงหน้า |
| ดูแลคนไข้ในวอร์ด (เวร IPD) | **Computer on Wheels (CoW)** / แท็บเล็ต | เข็นตามเตียง บันทึก vital / การให้ยา ข้างเตียง |
| จุดบริการต่าง ๆ | คอมที่ใช้ร่วมกัน (shared station) | login/logout บ่อย ต้องการเข้าระบบเร็ว |

> **Insight:** พี่ศิ "พิมพ์" มากกว่า "แตะ" และทำงานแบบมีคนรออยู่ตรงหน้าเสมอ — ทุกวินาทีที่ระบบหน่วงคือสายตาคนไข้ที่จ้องมา

---

### 3. Goals & Motivations

**เป้าหมายรายวัน:**
- กรอกข้อมูลคัดกรองให้ **ถูกต้อง ครบ และเร็ว** เพื่อให้คิวไหลลื่น ไม่ตกหล่น
- ส่งต่อข้อมูลให้หมอได้พร้อม หมอเปิดมาแล้วใช้งานได้เลย
- ไม่ให้คนไข้รอนาน / ไม่ให้เกิดข้อผิดพลาดที่กระทบความปลอดภัย
- ปิดงานบันทึกการพยาบาลได้ครบในเวร ไม่มีงานค้าง

**ความคาดหวังสูงสุดต่อ CIS:**
> "ขอแค่กรอกได้ไว ไม่สะดุด ฟอร์มอยู่ที่เดิมทุกครั้ง กดผิดแล้วแก้ง่าย — แล้วข้อมูลที่กรอกไปถึงหมอครบ ไม่หาย"

- **ฟอร์มกรอกเร็ว** — Tab/Enter ลื่น, ค่าที่กรอกบ่อยมี default/auto-fill, ไม่ต้องเลื่อนเมาส์ไปคลิกทีละช่อง
- **ตำแหน่งคงที่** — ปุ่มและช่องอยู่ที่เดิมทุกครั้ง (พึ่ง muscle memory ได้)
- **แก้ไขง่าย** — กดผิดแล้วย้อน/แก้ได้โดยไม่ต้องเริ่มใหม่
- **เห็นสถานะคิว** — รู้ว่าใครรออยู่ ใครเข้าหมอแล้ว

---

### 4. Pain Points & Frustrations

**สิ่งที่ทำให้หงุดหงิดที่สุด:**

- 🔴 **ฟอร์มยาว ช่องเยอะ บังคับกรอกเกินจำเป็น** — required field ที่ไม่จำเป็นทำให้ติดขัด ส่งคิวต่อไม่ได้
- 🔴 **กรอกซ้ำซ้อน** — ข้อมูลเดียวต้องกรอกหลายที่ เพราะแต่ละหน้าจอไม่คุยกัน
- 🔴 **flow เปลี่ยนหลังอัปเดต** — ปุ่มย้ายที่ ทำให้ muscle memory พัง ต้องเรียนรู้ใหม่ทั้งที่งานเร่ง
- 🔴 **กดผิดแล้วแก้ยาก** — เผลอกดยืนยันแล้วย้อนไม่ได้ ต้องตามแก้ทีหลัง
- 🔴 **ระบบ logout บ่อย / login ช้า** บน shared station — เสียเวลาสะสม
- 🔴 **ตัวหนังสือเล็ก สีเทากลมกลืน** — อ่านยากใต้แสงไฟวอร์ด / เมื่อรีบ

**Time Pressure & Cognitive Load:**
- มีเวลา **1.5–3 นาที/คน** กับคนไข้เป็นร้อย และมีคนรอจ้องอยู่ตรงหน้าตลอด
- ถูกขัดจังหวะถี่มาก (โทรศัพท์, คนไข้ถาม, เคสแทรก) → ต้องการระบบที่ "กลับมาทำต่อจากเดิมได้" ไม่หลุด context
- ความกลัวหลัก: **"กรอกผิด/ตกหล่นแล้วส่งต่อให้หมอผิด"** → ความเครียดสะสมจากการเป็นด่านหน้า

### 5. Specific Feature Interaction — AI Summary & Body Map

#### 🤖 ระบบสรุปการซักประวัติด้วย LLM

**ทัศนคติ:** *ตื่นเต้นแต่กังวลเรื่องความถูกต้องและการควบคุม*
> "ถ้ามันช่วยพิมพ์อาการที่คนไข้เล่าให้หน่อยก็ดีมากเลย มือฉันจะได้ว่างไปทำอย่างอื่น แต่ถ้ามันใส่อะไรที่คนไข้ไม่ได้พูด แล้วหมอเชื่อ — ฉันนี่แหละโดน"

**ความคาดหวัง:**
- ช่วยแปลงคำพูด/บทสนทนาคนไข้เป็นข้อความซักประวัติที่เรียบเรียงแล้ว (ลดภาระพิมพ์)
- ดึงค่าที่กรอกบ่อย (CC, อาการ, ระยะเวลา) มาเติมให้อัตโนมัติ แต่ยังให้เธอ **ตรวจและยืนยัน**
- ต้อง **ไม่ทำให้ flow ที่คุ้นเคยพังหรือช้าลง**

**ความกลัว:**
- 😨 **AI ใส่ข้อมูลที่คนไข้ไม่ได้พูด** แล้วเธอกลายเป็นคนรับผิดชอบ
- 😨 **เสียการควบคุม** — ระบบทำให้อัตโนมัติจนเธอไม่รู้ว่ากำลังบันทึกอะไรลงไป
- 😨 **แก้ข้อความ AI ยากกว่ากรอกเอง** บนฟอร์มที่เธอชินอยู่แล้ว
- 😨 ฟีเจอร์ใหม่ = **ต้องเรียนรู้ใหม่** ในวันที่งานล้นมือ

**สิ่งที่อยากเห็น (เพื่อความปลอดภัย + เร็ว):**
- ✅ AI เป็น **ผู้ช่วยเติม ไม่ใช่ผู้ตัดสินใจ** — เธอเห็นทุกอย่างก่อนยืนยันเสมอ
- ✅ **แยกชัด** ว่าส่วนไหน AI เติม ส่วนไหนเธอพิมพ์เอง (สี/ป้ายกำกับ)
- ✅ แก้ไขด้วย **คีย์บอร์ดล้วน** ได้ (Tab/Enter) ไม่บังคับให้ใช้เมาส์
- ✅ มี **undo / แก้ง่าย** และไม่ auto-submit
- ✅ flow ใหม่ **ทับซ้อนกับของเดิมให้น้อยที่สุด** เรียนรู้ได้ใน 1 ครั้ง

#### 🧍 Human Anatomy Indicator (Body Map แบบ Interactive)

**ทัศนคติ:** *ชอบที่มันสื่อสารชัดและลดการพิมพ์ แต่ห่วงเรื่องความเร็วในการกรอก*
> "เวลาคนไข้บอกว่าปวดตรงนี้ ถ้าฉันแตะที่รูปแล้วมันบันทึกให้เลยก็เร็วดี ดีกว่าพิมพ์ว่า 'ปวดบริเวณท้องน้อยด้านขวา' ทุกครั้ง"

**ความคาดหวัง:**
- ใช้เป็น **เครื่องมือกรอกอาการที่เร็วขึ้น** — แตะจุดบนร่างกาย → เลือกอาการ → จบ
- ช่วย **สื่อสารกับคนไข้** ได้ด้วย (ชี้ให้คนไข้ดูว่า "ตรงนี้ใช่ไหม")
- ผลที่ได้ต้อง **ส่งต่อให้หมออ่านง่าย** และตรงกับที่เธอบันทึก

**ความกลัว:**
- 😨 **ช้ากว่าพิมพ์** — ถ้าต้องซูม เล็งจุดเล็ก ๆ หลายที กว่าจะกรอกเสร็จคิวก็ยาว
- 😨 กดผิดจุด/ผิดข้าง บนเคาน์เตอร์ที่รีบ ๆ
- 😨 ต้องสลับมือระหว่างคีย์บอร์ด (ที่ถนัด) กับเมาส์/จอสัมผัสบ่อย ๆ ทำให้ flow สะดุด

**สิ่งที่อยากเห็น:**
- ✅ **เร็วกว่าหรือเท่าการพิมพ์** — แตะแล้วเลือกจาก list อาการที่พบบ่อย
- ✅ touch target ใหญ่ + รองรับทั้ง mouse และ touch (เพราะใช้หลายอุปกรณ์)
- ✅ ระบุ **ซ้าย-ขวา / หน้า-หลัง** ชัดเจน ป้องกันกรอกผิดข้าง
- ✅ บันทึกแล้ว **แสดงเป็นข้อความสรุป** ควบคู่ภาพ เพื่อให้ตรวจซ้ำและส่งต่อหมอได้
- ✅ ใช้งานได้โดย **ไม่ต้องละมือจากคีย์บอร์ดนานเกินไป**

## สรุปนัยเชิงออกแบบ (Design Implications)

ตารางเปรียบเทียบเพื่อใช้ตัดสินใจออกแบบและทำ Journey Map ต่อ

| มิติ | นพ. ธนกฤต (แพทย์) | คุณ ศิริพร (พยาบาล) |
|---|---|---|
| **โหมดหลัก** | อ่าน + ตัดสินใจ + สั่งการ | กรอก + คัดกรอง + ส่งต่อ |
| **อุปกรณ์เด่น** | แท็บเล็ต (ราวด์) + จอใหญ่ (OPD) | คอมเคาน์เตอร์ + คีย์บอร์ด + CoW |
| **อินพุตที่ถนัด** | touch + shortcut | keyboard (Tab/Enter) เป็นหลัก |
| **ต้องการสูงสุด** | ภาพรวมหน้าเดียว + สั่งของเร็ว | ฟอร์มเร็ว + ตำแหน่งคงที่ + แก้ง่าย |
| **กลัวที่สุด** | พลาดข้อมูลสำคัญ → คนไข้อันตราย | กรอกผิด/ตกหล่น → ส่งต่อหมอผิด |
| **ต่อ AI Summary** | ร่างให้แก้ + อ้างอิงได้ + ห้าม auto-save | เติมให้ + แยกชัด + แก้ด้วยคีย์บอร์ด |
| **ต่อ Body Map** | glanceable + ตรงข้าง = navigation | เร็วกว่าพิมพ์ + เลือกจาก list = input |

### หลักการออกแบบร่วม (ทั้งสอง Persona เห็นตรงกัน)

1. **เร็วคือฟีเจอร์** — ทุก interaction ต้องวัดเป็นวินาที เพราะคูณด้วยคนไข้หลักสิบ–ร้อยต่อวัน
2. **Human-in-the-loop เสมอ** — AI ร่าง/เติมได้ แต่ห้ามตัดสินใจหรือบันทึกเองโดยไม่มีการยืนยัน
3. **แยกให้เห็นว่าอะไรมาจาก AI** — ป้ายกำกับ/สีชัดเจน เพื่อความรับผิด (accountability) และความไว้ใจ
4. **ความถูกต้องเชิงคลินิกต้องมาก่อนความสวย** — โดยเฉพาะ ซ้าย-ขวา, ตัวเลข, ขนาดยา
5. **ความสม่ำเสมอ (consistency)** — อย่าย้ายปุ่ม/เปลี่ยน flow โดยไม่จำเป็น muscle memory คือสินทรัพย์
6. **อ่านได้แม้รีบ** — typography ใหญ่พอ, contrast พอ, ลด clutter, glanceable
7. **กดผิดต้องกู้คืนได้** — undo / แก้ inline / ไม่ auto-submit ในจุดที่กระทบความปลอดภัย

### ข้อเสนอสำหรับขั้นต่อไป (Next Steps)
- นำ Persona ทั้งสองไปทำ **User Journey Map** แยกตามบริบท: ราวด์ IPD (แพทย์) และ คัดกรอง OPD (พยาบาล)
- ทำ **Empathy Map** เพื่อขุด emotional state ในแต่ละ touchpoint
- ใช้ Pain Points เป็น input ของ **How Might We** ในขั้น Ideation
- ทดสอบสมมุติฐานเรื่อง AI Summary และ Body Map ด้วย **usability testing** กับผู้ใช้จริงในสองบทบาทนี้
